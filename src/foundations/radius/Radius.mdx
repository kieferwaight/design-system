# Radius Foundation

The design system maintains two distinct, deliberately separate corner-radius systems to handle different visual hierarchies:

1. **Layer-Based Radii** organize concentric **nested containers**.
2. **Glyph Radii** pair a **single component's** corners to its physical size so it reads as a rounded square at any scale.

---

## 1. Layer-Based Radii (Nested Containers)

When containers are nested inside one another, utilizing a single, constant corner radius breaks visual concentricity (causing the corners to "pinch" or look awkward). 

To solve this, we enforce the **concentricity rule**:
$$\text{inner-radius} = \text{outer-radius} - \text{gap}$$

Our layer scale assumes the baseline `--space-1` gap, descending from the outermost container (e.g. desktop window or iOS screen) to the innermost element:

## 2. Glyph Radii (Size-Paired Corner Scale)

Visual atoms (such as Swatches, Avatars, or Chips) read as rounded squares at any scale. If a constant corner radius is used, small boxes round too much (turning into circles) while large boxes don't round enough (turning into sharp squares).

Glyph corner radii are paired directly to the box's size via a dedicated ladder:

### Dynamic Interpolation

For arbitrary size values between or beyond the defined buckets, the `glyphRadius(sizePx)` utility interpolates these anchors cleanly to ensure perfect optical proportions at any custom scale.
